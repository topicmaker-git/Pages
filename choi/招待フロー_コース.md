# コースへのメンバー招待フロー

対応ユースケース: UC-15(コースメンバー招待)、加えて被招待者側フロー(団体招待の UC-50/51/52 と類似)

## 重要な前提: コースメンバーの団体所属制約

**コースメンバー(共同編集者)は、コース主催者と同じ団体に所属する会員のみ。**

- コース主催者は団体所属を前提とする(無所属会員はコースを作らない)
- コースメンバーも主催者と同じ団体に所属する必要がある
- 団体管理者が介入してメンバーを招待する場合も、同じ制約が適用される(上位権限としても他団体・無所属の会員は招待できない)

これにより、コースは主催者の所属団体内における共同作業の場として閉じる。

## 招待状態の遷移

```mermaid
stateDiagram-v2
    [*] --> 送信済: メール招待または自団体会員指名で送信
    [*] --> URL発行済: 招待URL方式で発行
    送信済 --> 承諾済: 被招待者が承諾
    URL発行済 --> 承諾済: 自団体内会員がURLから承諾
    送信済 --> 辞退: 被招待者が辞退
    送信済 --> 期限切れ: 有効期限を経過
    URL発行済 --> 期限切れ: 有効期限を経過
    期限切れ --> [*]: 主催者/管理者がレコード削除
    承諾済 --> [*]: コースメンバーとして正式参加
    辞退 --> [*]: 主催者/管理者がレコード削除
```

## A-CM1 コースメンバー一覧での招待状態の表示

| 招待状態 | A-CM1 での表示 | 可能な操作 |
|---|---|---|
| 送信済 | 氏名 or メール + 「招待中」バッジ | 招待取消 / 再送信 |
| URL発行済 | 「URL発行中(使用状況不明)」バッジ | URL無効化 |
| 承諾済 | 通常のメンバーレコード | 除外 |
| 期限切れ | 氏名 or メール + 「期限切れ」バッジ | 再招待 / レコード削除 |
| 辞退 | 氏名 or メール + 「辞退」バッジ | レコード削除 |

## 団体招待フローとの差分

| 項目 | 団体招待(UC-03) | コース招待(UC-15) |
|---|---|---|
| 招待方法 | メール / 既存ID指名 / CSV一括 | 自団体会員選択 / メール / URL発行 |
| 被招待者の所属制約 | 別団体所属者は招待不可(1会員1団体) | **主催者と同じ団体所属の会員のみ**(他団体・無所属は招待不可) |
| 招待元の権限 | 団体管理者のみ | 主催者本人 または 団体管理者 |
| 承諾時の効果 | Account.所属団体 / 団体ロール がセット | コース o-- Account メンバー関係が生成 |
| URL発行方式 | なし | あり(自団体内のみ承諾可) |
| 「自団体会員から選択」方式 | — | あり(主催者・管理者ともに使用可) |

## 重要な設計ポイント

1. **コースメンバーは主催者と同じ団体所属の会員のみ**
   - 自団体内会員のみ招待可能
   - 他団体所属者・無所属会員は招待不可
   - 団体管理者の上位権限としても、この制約は外れない
   - 動機: コースは主催者の所属団体内での共同作業の場として閉じる

2. **招待URL方式の特殊性**
   - メール経由の招待と違い、被招待者が**事前に確定していない**
   - A-CM1 には「URL発行中」として1レコードのみ表示する(何人が受諾するか不明)
   - URL を受け取った人が**自団体会員ならば承諾可、それ以外はエラー**
   - URL1つにつき1人の承諾で終わらせるか、複数人承諾を許すかは設計判断(下の未解決事項参照)

3. **自団体会員選択方式**
   - ドロップダウンから自団体内の会員を選んでタグとして追加
   - 主催者本人・団体管理者ともに使用可
   - 内部的にはメール招待と同じトークン発行フローを踏むが、相手が特定できているので、招待メール内のリンクでログインさせる設計も可能(自団体会員は団体所属時にちょい英語アカウントを持つ)

4. **団体管理者が介入してメンバーを招待する場合**
   - 主催者本人と同じ範囲(主催者と同じ団体内の会員のみ)で招待可能
   - 団体管理者の介入編集権限は、主催者と同じ範囲で招待操作を行えるようにするものであり、招待範囲を拡張するものではない

## 設計判断が必要な事項

1. **招待URL方式の使い回し**: URL 1つにつき 1人の承諾で終了(ワンタイム)か、複数人(自団体内会員)承諾可能か
2. **URL方式の発行数上限**: 無制限に発行可能か、コースあたり N 個までか
3. **自団体会員選択方式のメール送信**: メール送信をスキップして、対象会員に「コース招待通知一覧」画面を見せる設計も可能
4. **被招待者が既にコースメンバーだった場合のフェイルセーフ**: 招待発行時だけでなく承諾時にも重複チェック
5. **投稿者制限との整合**: コースの投稿者制限が「一般(メンバー以外も投稿可)」の場合、コース招待の意義は「公式メンバー化」の意思表示。UI に説明を入れるべきか
6. **招待URL方式で来た場合のアカウント作成画面**: メール情報がないので、完全な新規アカウント作成フロー。新規アカウント作成後、団体所属がない状態となるため、URL承諾フローでは「先に団体に所属してから再アクセス」という案内になる可能性。実装時に詳細設計が必要
7. **新規アカウント作成 → 団体未所属 → コース招待エラー** という連鎖の UX 改善方法

## 本フェーズ対象外 / 将来拡張

- 招待URLの QR コード発行
- 招待URLのアクセス解析(何人が見たか、承諾率など)
- 大量招待時のレート制限
- 主催者不在時の主催者再選定(現主催者退職等のケース)

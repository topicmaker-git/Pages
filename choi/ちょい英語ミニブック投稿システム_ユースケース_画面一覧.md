## システム前提

本システムの管理系画面(A画面群)は、すべての団体が共通で利用できる
(各団体が自団体の範囲で管理を行う)。

ちょい英語サービスの運営会社である株式会社トピックメーカーは
システム内で唯一の特別な立場にあり、同社が主催するコースは、
学習者向けの既存B画面(B-01ホーム等のコース一覧)に掲載される
対象となる。各団体がそれぞれ主催するコースを自団体メンバー向けに
一覧表示するローカルなコース一覧(学習者向け)画面は、本設計の
スコープ外である。

トピックメーカーは団体モデルの`isトピックメーカー`属性(boolean)で
識別される(true の団体はシステム内で唯一)。

### 会員と団体の関係

- 1人の会員は最大1つの団体にのみ所属する(複数団体所属は不可)
- 団体に所属しない会員(無所属会員)も存在する
- 会員は団体に所属するかどうかに関わらず、ちょい英語アカウントを
  保持してログインできる

## アクター（会員のロール）

| ロール | 説明 |
|--------|------|
| 団体管理者 | 団体の登録・管理、メンバー管理を行う会員。Account.団体ロール=管理者 |
| コース主催者 | コースの作成・設定・管理を行う会員。コースに対する主催関係で定義される |
| コースメンバー | コースに参加し、ユニットを投稿できる会員。コースに対するメンバー関係で定義される |
| 投稿者 | ユニットの投稿を行う会員。次のいずれか: (a) コース主催者、(b) コースメンバー、(c) 投稿者制限が「一般」に設定されたコースにユニットを投稿する任意の会員、(d) コース非帰属ユニットを投稿する任意の会員 |
| 学習者 | 投稿された教材を学習する会員 |

※ 1人の会員が複数ロールを兼ねうる
※ 「投稿者」「学習者」は独立した属性ではなく、会員がユニット投稿または学習を行う際の関係性として現れるロール
※ アカウント管理（登録・ログイン・プロフィール編集等）は実装済のため省略

## 公開範囲の定義

コース・ユニットの公開設定は以下の4値を取る:

| 公開レベル | 意味 |
|---|---|
| 公開 | 全ユーザーに公開 |
| プライベート公開 | URLを知っている会員のみアクセス可能 |
| 団体限定公開 | 指定された団体に所属する会員のみアクセス可能 |
| 非公開 | 投稿者本人のみアクセス可能 |

**コース帰属ユニットの公開範囲**:
コース帰属ユニットの実効的な公開範囲は、所属コースの公開設定に従う。ユニット自身の公開設定は「公開 / 非公開」のみをUIで選択可能とし、コースの公開範囲内で当該ユニットを公開するか否かを決定する。

**コース非帰属ユニットの公開範囲**:
ユニット自身が4値すべてを選択可能。

### ユニットの公開属性のデータ保持

ユニットの`公開`属性はモデル上は常に4値enum(公開 / プライベート公開 / 団体限定公開 / 非公開)として保持する。UIでの選択可能な値はコース帰属状態に応じて2値または4値に絞られるが、DB層では常に4値のいずれかが格納される。

### コース帰属状態の変更時の公開値の扱い

ユニットのコース帰属状態が切り替わる際の`公開`値は、以下のルールに従う。

**コース非帰属 → コース帰属 への切替時**:

- ユニットの公開値が「非公開」の場合: コース側の公開値にかかわらず「非公開」のまま不変
- ユニットの公開値が「公開 / プライベート公開 / 団体限定公開」のいずれかの場合:
  - コース側の公開値と一致していれば不変
  - 不一致であればコース側の公開値でユニットの公開値を強制上書きする

具体例:

| 切替前のユニット公開値 | コースの公開値 | 切替後のユニット公開値 |
|---|---|---|
| 非公開 | (任意) | 非公開(不変) |
| 公開 | 公開 | 公開(不変) |
| 公開 | プライベート公開 | プライベート公開(上書き) |
| 公開 | 団体限定公開 | 団体限定公開(上書き) |
| プライベート公開 | 団体限定公開 | 団体限定公開(上書き) |
| 団体限定公開 | 公開 | 公開(上書き) |

上書きが発生する場合、UI側では保存前にユーザーに確認を行うことが望ましい(警告モーダル等)。

**コース帰属 → コース非帰属 への切替時**:

- DBに保持されている公開値をそのまま引き継ぐ
- UIは4値ラジオに切り替わり、現在のユニット公開値を初期選択として表示する

**新規投稿時(A-MU3)**:

- 切替前の値が存在しないため、コース帰属を選択した時点でUIは2値(公開/非公開)のみを表示し、ユーザーが選択した値を保存する

## ユースケース一覧

### 1. 団体管理（団体管理者）
- UC-01: 団体を登録する
  - 前提: ちょい英語のアカウントでログイン済みの会員が実行
  - 前提: 実行会員は団体未所属であること(1会員は1団体のみ所属)
  - 事後条件: 登録された団体が作成され、実行会員がその団体の
    デフォルト管理者となる(Account.団体ロール=管理者)
  - 管理者権限は後でUC-04により変更可能(招待した他の会員を
    管理者にすることもできる)
- UC-02: 団体情報を編集する（団体名、管理部署、電話）
- UC-03: 会員を招待・追加する
  - 招待対象は団体未所属の会員のみ(1会員は1団体のみ所属の
    制約による)
- UC-04: 会員のロールを変更する（管理者 / 一般）
- UC-05: 会員を削除する

### 1b. トピックメーカー識別フラグの設定（システム運用、スコープ外）
- UC-06: `isトピックメーカー`属性を設定する
  - 実行主体: 株式会社トピックメーカーの運用担当者
  - 実行方法: 本システムのUI上には設定画面を設けず、
    DB直接操作またはマイグレーションスクリプト等の
    運用ツールで行う
  - 制約: システム全体で true を持つ団体は最大1つ
  - **本設計のスコープ外** (画面設計対象ではない)

### 2. コース管理（コース主催者）
- UC-10: コースを作成する
- UC-11: コース情報を編集する(コース名、紹介、紹介ビデオ)
- UC-12: コースの公開設定を変更する(公開 / プライベート公開 / 団体限定公開 / 非公開)
  - 「団体限定公開」選択時は、対象団体を指定する
- UC-13: 投稿者制限を設定する(主催者のみ / メンバー / 一般)
  - 「一般」: コースメンバー以外の任意の会員も投稿可能
- UC-14: コースの制約値(最大ユニット数、最大デッキ数、デッキ内最大カード数)を参照する
  - 本項目はコース主催者が設定するものではなく、システムが
    コースの所属文脈(団体への帰属、isトピックメーカーか、
    将来的には課金プランや法人契約等)に応じて自動的に設定する
  - コース作成時にシステムが値を決定してコース属性にセットし、
    以降は変更不可(モデル属性としては保持、UIでは情報提示のみ)
  - コース主催者・団体管理者はこの値を画面上で**参照**できる
- UC-15: コースにメンバーを招待する
- UC-16: コースの表示順を変更する
- UC-17: コースを削除する

### 2b. 自団体コースの集約管理（団体管理者）
- UC-18: 自団体メンバーが主催するコースを一覧・確認する
  - 主アクター: 団体管理者
  - 対象: 自団体所属会員が主催するすべてのコース
  - 閲覧のみで、主催者本人のマイコース一覧とは別画面として提供
- UC-19: 自団体メンバーが主催するコースを編集・削除する
  - 主アクター: 団体管理者
  - 内容: UC-11〜14, UC-16, UC-17 と同等の編集・削除操作を、
    他メンバーが主催するコースに対しても実行可能(上位権限)
  - 介入編集の履歴ログは将来拡張点(本フェーズ対象外)

### 3. ユニット投稿（投稿者）
- UC-20: ユニットを新規作成・投稿する
    - ユニット定義（ユニット名、紹介、カテゴリー、サブカテゴリー、フリータグ、画像、公開設定、料金設定）
    - エクセルファイルのアップロード
    - 音声パック（zip）のアップロード（任意）
    - 画像パック（zip）のアップロード（任意）
    - コース非帰属で投稿する場合、公開設定は4値から選択
    - コース帰属で投稿する場合、公開設定は「公開 / 非公開」から選択
- UC-21: ユニット情報を編集する
- UC-22: 投稿データを更新する（差分/全置換）
    - 投稿データ(エクセルファイル)内の行データの一部または全部を編集する
    - エクセルファイル自体のバージョン管理は行わない
- UC-23: 投稿データをダウンロードする
- UC-24: ユニットをコースに紐づける
    - 1つのユニットは高々1つのコースにのみ紐づく
    - コース帰属/非帰属の切替時のユニット公開値の扱いは「公開範囲の定義」セクションのルールに従う
- UC-25: ユニットの表示順を変更する
- UC-26: ユニットを削除する

### 3b. 自団体ユニットの集約管理（団体管理者）
- UC-27: 自団体メンバーが投稿したユニットを一覧・確認する
  - 主アクター: 団体管理者
  - 対象: 自団体所属会員が投稿したすべてのユニット
  - 閲覧のみで、投稿者本人のマイユニット一覧とは別画面として提供
- UC-28: 自団体メンバーが投稿したユニットを編集・削除する
  - 主アクター: 団体管理者
  - 内容: UC-21〜26 と同等の編集・削除操作を、他メンバーが投稿した
    ユニットに対しても実行可能(上位権限)
  - 介入編集の履歴ログは将来拡張点(本フェーズ対象外)

### 4. コンテンツ閲覧・学習（学習者）
- UC-30: コース一覧を閲覧する
- UC-31: コース内のユニット一覧を閲覧する
- UC-32: ユニットを検索する（カテゴリー、サブカテゴリー、フリータグ）※検索条件は別途確認
- UC-33: デッキを検索する（フリータグ）
- UC-34: デッキでフラッシュカード学習をする

### 5. お気に入り・コメント（学習者）
- UC-40: デッキをお気に入りに登録する
- UC-41: お気に入りを解除する
- UC-42: お気に入りしたデッキの一覧を見る
- UC-43: デッキにコメントを投稿する
- UC-44: コメントを削除する
- UC-45: デッキのコメント一覧を閲覧する

※ お気に入り・コメントの対象はデッキのみ（ユニット・カード単位は対象外）

## 補足事項

- **パートの扱い**: 既存「ちょい英語」のデータ構造との互換性のため、ユニット内部にパート層を持つが、「ちょい英語ミニブック」では1ユニット=1パート固定とし、UIには露出しない
- **料金設定**: 課金システムは既存「ちょい英語」を利用。ユニット側は`無料 / 有料`フラグのみ保持
- **投稿データのバージョン管理**: エクセルファイル自体のバージョン管理は行わない（常に最新版のみ保持）
- **コース・ユニットの所有関係**: コース・ユニットは会員個人が主催/投稿する(団体の所有物ではない)。団体所属の有無に関わらず、会員はコース・ユニットを作成可能。団体管理者は自団体メンバーのコース/ユニットに対して集約管理ビュー(団体〇〇)からアクセスでき、編集・削除などの上位権限を持つ(UC-18, 19, 27, 28)
- **介入編集の履歴**: 団体管理者が他メンバーのコース/ユニットに介入編集を行った際の履歴ログは、本フェーズ対象外(将来拡張点)
- **フェーズ1運用**: 本設計はトピックメーカー社内ドッグフーディング運用を前提とする。社員が各種ロール(無所属/団体一般/団体管理者)のアカウントを使い分けて、フェーズ2で想定される全ユーザー体験を事前検証する。アクセス入口はトピックメーカー専用暫定プロフィール画面(A-P1、URL直接入力)

## ディスカッション(未解決/要検討事項のメモ)

### コースの制約値(最大ユニット数、最大デッキ数、デッキ内最大カード数)の扱い

**現在の方針**:
- モデル上はコース属性として保持する(概念モデル図のまま)
- コース作成時にシステムが所属文脈(団体への帰属、isトピックメーカーか、
  将来的には課金プランや法人契約等)から値を算出して自動セットする
- 作成後は変更不可(UIから編集できない)
- コース編集画面には**情報提示のみ**として表示する(参照のみ)

**この方針の根拠**:
- 課金や法人契約によってさまざまな制約が将来想定されるため、モデル上は
  コース個別に値を保持できる柔軟性を残す(コース作成時のスナップショット)
- 主催者が自由に設定できる値ではないことをUI上で明確化する

**今後検討が必要な事項**:
- 制約値の決定ロジックの詳細設計(どの属性からどう算出するか)
- プラン変更などにより後から制約値を更新する必要が生じた場合の対応
  (例: 団体がプランアップ → 既存コースの上限も緩和するか否か)
- デフォルト値(無所属会員の場合など、所属団体情報から決まらないケース)
- 制約に達した際のユーザーへのフィードバック(エラー表示の文言、
  プランアップ導線等)

### 団体への招待を受けた側のユースケース/画面(未設計)

**現状の課題**:
UC-03「会員を招待・追加する」は団体管理者側の操作として記述されているが、
**招待される側**のユースケースと画面フローが抜けていた。招待メールが
届いてから団体所属が完了するまでの一連のフローを設計する必要がある。

**分岐の軸**:
招待を受けた時点で、被招待者が既にちょい英語アカウントを持っているか
否かでフローが変わる。

**フロー案(要設計)**:

1. 被招待者が既にちょい英語アカウント保持者の場合
   - 招待メール内のリンクをクリック
   - ログイン(未ログインの場合)
   - 招待承諾画面が表示され、団体情報と招待内容を確認
   - 承諾すると当該団体に所属(Account.所属団体がセットされる)
   - 既に別団体に所属している場合は招待を受けられない旨のエラー

2. 被招待者がちょい英語アカウント未保持の場合
   - 招待メール内のリンクをクリック
   - アカウント新規作成画面に誘導(メールアドレスは招待メール宛先で
     プリフィルしておく)
   - アカウント作成完了後、自動的に当該団体に所属した状態になる
   - または、作成後に改めて招待承諾画面を表示して確認を取る

**設計判断が必要な事項**:
- 招待メールの発行タイミング・有効期限
- 招待リンクのセキュリティ(トークン、ワンタイム性、期限切れ時の挙動)
- 招待承諾画面を本システム(管理コンソール)側で実装するか、
  既存ちょい英語アカウント系システム側で実装するか(責任分界点)
- 招待に対する「辞退」「ブロック」等の操作の要否
- 被招待者が既に別団体所属の場合の扱い(単純エラー? 所属切り替え機能?
  — 現状仕様では1会員=1団体制約があるので切り替えが必要な場合は
  旧団体からの脱退が先に発生する)
- 招待を送った後の状態管理(招待中/承諾済/期限切れ/辞退など)と、
  A-M1会員一覧画面での表示(招待中の会員を一覧に含めるか、別扱いか)

**関連する追加ユースケース(暫定UC番号)**:
- UC-50: 団体への招待メールを受け取って承諾する(アカウント既存)
- UC-51: 団体への招待メールを受け取ってアカウント作成と承諾を行う
- UC-52: 招待状態を管理する(団体管理者側、A-M1での表示)

コースメンバー招待(UC-15)についても同様の被招待者フローが必要になる
可能性が高い。設計時に揃えて検討する。

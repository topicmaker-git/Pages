# ちょい英語ミニブック投稿システム — 投稿データ / メディアパック仕様

本書は、ユニット投稿(UC-20)および投稿データ更新(UC-22)で扱う、
投稿データ(エクセル) / 音声パック / 画像パック の構造と処理仕様を定める。

## 1. エクセル構造

### 1.1 シート構成

- シート数: 1(シート名は任意、実装上は先頭シートのみを読む)
- 1行目: ヘッダー行
- 2行目以降: データ行(1行=1センテンス)

### 1.2 列定義

| # | 列名 | 対応する概念モデル属性 | 記入粒度 | 必須 |
|---|---|---|---|---|
| 1 | チャプター名 | デッキ(識別キー) | 各デッキの全行に同じ値 | 必須 |
| 2 | シーン名 | カード(識別キー) | 各カードの全行に同じ値 | 必須 |
| 3 | 話者 | センテンス(表示用情報) | 各行固有 | 任意 |
| 4 | 英会話 | センテンス.英文 | 各行固有 | 必須 |
| 5 | 手順3で作成した語順訳 | センテンス.直訳 | 各行固有 | 任意 |
| 6 | 和訳 | センテンス.和訳 | 各行固有 | 任意 |
| 7 | チェック | 投稿者側の作業フラグ(DBには保存しない) | 各行固有 | 任意 |
| 8 | 音声ファイル | センテンス.音声ファイルID に相当 | 各行に記入(1発話=1音声) | 条件付き |
| 9 | デッキ画像 | デッキ.画像ファイルID に相当 | **デッキの先頭行のみ**記入、他行は空欄 | 条件付き |
| 10 | カード画像 | カード.画像ファイルID に相当 | **カードの先頭行のみ**記入、他行は空欄 | 条件付き |

列8〜10 の「条件付き」は、TTS/AI画像の自動生成が ON の場合は空欄可、
OFF の場合は必須となる(詳細は §3)。

### 1.3 階層の識別ルール

- **デッキの識別**: 「チャプター名」の値が変わる境界で新しいデッキが始まる
- **カードの識別**: 「シーン名」の値が変わる境界で新しいカードが始まる
- **センテンスの識別**: データ行1行 = 1センテンス
- **デッキ・カードの表示順**: エクセル内での登場順をそのまま表示順とする

### 1.4 デッキ画像・カード画像の先頭行ルール

```
チャプター名 | シーン名 | ... | デッキ画像 | カード画像
チャプター1  | シーン1  | ... | c1.png    | c1s1.png     ← デッキ先頭 かつ カード先頭
チャプター1  | シーン1  | ... |           |              ← 2行目以降は空欄
チャプター1  | シーン1  | ... |           |              ← 同上
チャプター1  | シーン1  | ... |           |              ← 同上
チャプター1  | シーン2  | ... |           | c1s2.png     ← カード先頭(デッキは継続)
チャプター1  | シーン2  | ... |           |              
...
チャプター2  | シーン1  | ... | c2.png    | c2s1.png     ← デッキが切り替わる
```

- 先頭行以外の行に画像ファイル名が書かれていた場合は**バリデーションエラー**
- 先頭行が空欄だった場合は「このデッキ/カードに画像なし」と解釈

### 1.5 ファイル名の命名規則

投稿者の自由(拡張子と S3 に許容される文字集合のみ強制)。

- 許容文字: 英数字、アンダースコア(`_`)、ハイフン(`-`)、ドット(`.`)
- 禁止: 空白、日本語、スラッシュ、その他特殊文字
- 音声: `.mp3` / `.wav` / `.m4a` 等(実装で定義)
- 画像: `.png` / `.jpg` / `.jpeg` / `.webp` 等
- 動画(将来): `.mp4` 等

同じファイル名を複数行(複数センテンス/複数カード)が参照しても良い。
その場合、S3上には1つのオブジェクトのみ存在し、DB上の複数レコードが
同じキーを参照する。

## 2. zip構造

### 2.1 音声パック

```
audio.zip
├── c1_s1_u1.mp3
├── c1_s1_u2.mp3
├── c1_s1_u3.mp3
├── ...
└── c2_s8_u4.mp3
```

- フラット構造(サブディレクトリなし)を原則とする
- ファイル名はエクセルの「音声ファイル」列と完全一致
- zip内で重複するファイル名は不可

### 2.2 画像パック

```
images.zip
├── c1.png              ← デッキ画像
├── c2.png              ← デッキ画像
├── c1s1.png            ← カード画像
├── c1s2.png
├── ...
└── c2s8.png
```

- フラット構造を原則
- デッキ画像とカード画像は同一zip内に混在して良い
- ファイル名はエクセルの「デッキ画像」「カード画像」列と完全一致

### 2.3 動画の扱い(将来拡張)

画像パックに動画ファイルを含める形を想定するが、本フェーズでは対象外。
`画像パック.メディアタイプ : 静止画 / 動画` enum との対応は、将来
「画像パック」を「メディアパック」に概念拡張するタイミングで整備する。

## 3. 自動生成モードの挙動

A-MU3 新規投稿画面には以下のチェックボックスがある:

- **音声**: 「アップロードの代わりに TTS で自動生成する」
- **画像**: 「デッキに自動で画像を割り当てる(画像自動)」

### 3.1 音声自動生成 ON の場合

- エクセルの「音声ファイル」列は**空欄可**(空欄=自動生成対象)
- 投稿者が音声zipを同時にアップロードすることも可能
- この時:
  - エクセルで**ファイル名が書かれた行**は、zip内の該当ファイルが必須
  - エクセルで**空欄の行**は、TTS で自動生成される
- システムは自動生成した音声を S3 に格納し、システム生成ファイル名を
  付与する(例: `_gen_sent_00001.mp3` のような予約プレフィクス)

### 3.2 画像自動生成 ON の場合

- エクセルの「デッキ画像」「カード画像」列は**空欄可**
- 画像zipを同時にアップロードすることも可能(混在許容)
- 空欄の行については AI 画像生成で自動補完

### 3.3 自動生成 OFF の場合

- エクセルの該当列は必須
- zipに該当ファイルがなければエラー(§4 参照)

## 4. バリデーション

A-MU3 投稿時 / A-MU2 差分適用時に実施する検証。

### 4.1 構造チェック(エクセル単体)

- 必須列(チャプター名、シーン名、英会話)が全て埋まっているか
- デッキ画像・カード画像が先頭行以外に書かれていないか(§1.4)
- ファイル名に禁止文字が含まれていないか(§1.5)

### 4.2 整合性チェック(エクセル × zip)

**不整合はエラーとして投稿を拒否する**(自動補完は行わない)。

不整合と判定されるケース:
1. エクセルで参照されているファイル名が zip にない
   - 例外: 自動生成モード ON 時のエクセル空欄行は対象外
2. zip にあるがエクセルから参照されていない余剰ファイル
3. 同じファイル名が zip 内で重複
4. ファイル拡張子が許容リスト外

エラー時の UI 表示例(既存ワイヤーフレームの警告バー):

> エラー:
> - 音声ファイル `c1_s1_u1.mp3` がエクセルで参照されていますが、zipに見つかりません
> - 画像ファイル `extra.png` がzipに含まれていますが、エクセルから参照されていません
> - デッキ「チャプター2」のカード画像がシーン先頭行ではない位置(行15)に記載されています

### 4.3 スキーマ制約チェック(UC-14 連携)

- 最大デッキ数、最大カード数、デッキ内最大カード数 を超過していないか
- 超過時はエラーで投稿拒否

## 5. S3 格納規則

### 5.1 キー構造

```
s3://{bucket}/
  {ユーザーID}/
    {ユニットID}/
      {投稿者が命名したファイル名}
```

- `ユーザーID`: Account の安定識別子(ULID等)
- `ユニットID`: Unit の安定識別子
- ファイル名はエクセルで指定されたものをそのまま使う

### 5.2 ユニットID の発行タイミング

- **A-MU3「投稿する」ボタン押下時**にユニットIDが確定する
- それまではS3への保存は発生しない
- したがって A-MU3 では**エクセル + (必要なら)zip の一括アップロード**が必須
- 下書き保存も「投稿する」と同様に扱い、ユニットIDを発行する
  (公開設定を「非公開」にするだけの違い)

### 5.3 ユニット表紙画像

A-MU3/A-MU2 の UI 上で**エクセルとは別に**直接アップロードする
(現行ワイヤーフレームの「カバー画像」フィールド通り)。S3キーは:

```
s3://{bucket}/{ユーザーID}/{ユニットID}/_cover.{ext}
```

アンダースコア始まりの予約ファイル名を使い、投稿者が同名のファイルを
エクセルから参照していた場合は警告(ただしこの予約ファイル名は
投稿者には公開しないので、通常衝突しない)。

### 5.4 UC-22 差分更新時の挙動

- ユニットIDは既に確定しているので、S3キーは維持される
- エクセルの「音声ファイル」列の値を変更した場合:
  - 旧ファイル名が他のセンテンスから参照されなくなっていれば、
    S3から削除(ガーベジコレクション)
  - 新ファイル名が zip に含まれていれば S3 に追加アップロード
- ファイル内容の差し替え(ファイル名は同じ、中身だけ更新):
  - 同じ S3 キーに上書き
- 差分プレビューでこれらの変化を投稿者に表示する

## 6. 処理フロー(A-MU3 新規投稿)

```
1. 投稿者がエクセル(+必要ならzip)をアップロード
   ↓
2. システムがエクセルをパース
   ↓
3. 構造チェック(§4.1)
   ↓ NG → エラー表示、中断
   ↓ OK
4. zipを展開(zipがある場合のみ)
   ↓
5. 整合性チェック(§4.2)
   ↓ NG → エラー表示、中断
   ↓ OK
6. スキーマ制約チェック(§4.3)
   ↓ NG → エラー表示、中断
   ↓ OK
7. 投稿プレビュー表示(投稿者が確認)
   ↓
8. 投稿者が「投稿する」クリック
   ↓
9. ユニットIDを発行
   ↓
10. エクセル内容をDBに永続化
    - ユニット → パート(1個固定) → デッキ → カード → センテンス
    ↓
11. zip内のメディアをS3に転送
    ↓
12. 自動生成対象(空欄行)があればTTS/AI画像生成を非同期ジョブに投入
    ↓
13. A-MU2 に遷移(ユニット詳細/編集画面)
```

## 7. 処理フロー(A-MU2 差分更新)

```
1. A-MU2 画面で「⭳ 投稿データをダウンロード」クリック
   ↓
2. システムが現在のDB内容をエクセル形式で出力
   - 音声ファイル列、デッキ画像列、カード画像列は現在のS3キーを反映
   ↓
3. 投稿者がエクセルを編集(行の追加/変更、ファイル名の変更等)
   ↓
4. A-MU2 で編集後エクセル(+必要ならzip)をアップロード
   ↓
5. システムが差分を解析
   - センテンス単位の create / update / ignore を判定
   - ファイル名列の変化からS3操作(追加/削除/上書き)を計算
   ↓
6. バリデーション(§4)
   ↓ NG → エラー、中断
   ↓ OK
7. 差分プレビュー表示
   - 変更サマリ(例: create 3 / update 12 / delete 1 / ignore 50)
   - S3 操作サマリ(例: 音声追加 3 / 削除 1 / 差し替え 2)
   ↓
8. 投稿者が「差分を適用」クリック
   ↓
9. DB 更新 & S3 操作 実行
   ↓
10. 完了
```

## 8. 既存データ例との対応

参考データ: 「インバウンド会話_higoアカウント_0422版.xlsx」

- **ユニット**: 「インバウンド会話」(ファイル名から類推)
- **パート**: 1個(固定、UI非露出)
- **デッキ**: 2個(チャプター1:日本到着と街歩き、チャプター2:日本文化と旅の締めくくり)
- **カード**: 16個(各デッキ内に8シーン)
- **センテンス**: 64個(各カード内に4発話)

チャプター1:日本到着と街歩き | シーン1:空港の案内所で... | 案内係   | Thank you, where...| ... | 0 | c1s1u3.mp3 |        |
チャプター1:日本到着と街歩き | シーン1:空港の案内所で... | スタッフ | You can buy...     | ... | 0 | c1s1u4.mp3 |        |
チャプター1:日本到着と街歩き | シーン2:ホテルの受付で... | 旅行者   | Good afternoon... | ... | 0 | c1s2u1.mp3 |        | c1s2.png

zipに含まれるファイル:
- 音声zip: 64個のmp3(c1s1u1.mp3 〜 c2s8u4.mp3)

## 9. 残論点 / 将来拡張

1. **ユニット表紙画像のS3キー衝突対策**: `_cover.{ext}` の予約名について、投稿者への公開範囲と衝突検知の実装詳細
2. **音声/画像の上限サイズ**: 1ファイルあたりの最大サイズ、zipの最大サイズ
3. **S3オブジェクトのライフサイクル**: 非公開ユニットのメディアへの署名付きURLの発行、期限管理
4. **自動生成の非同期処理**: 投稿時に自動生成が完了しない場合、ユーザにどう見せるか(「生成中」バッジ等)
5. **動画パック**: 将来的な拡張として概念モデルの `画像パック` を `メディアパック` に拡張
6. **ファイル名の大文字小文字**: S3キーは大文字小文字を区別するが、エクセルでの入力揺れ(Windows感覚で小文字化しがち)への対応
7. **zipのバージョン管理**: 投稿データ(エクセル)はバージョン管理しない方針(UC-22注記)だが、メディアについても同様とするか
